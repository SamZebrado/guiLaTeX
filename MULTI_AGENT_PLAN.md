# guiLaTeX 多Agent开发计划

## 项目概述
- **项目名称**: guiLaTeX
- **项目描述**: 可视化LaTeX编辑器，支持像Photoshop那样选中拖动编辑LaTeX元素，像Word那样改变字体字号颜色排版
- **技术栈**: PyQt6 (Python), PyMuPDF, TeX Live 2022
- **当前阶段**: Phase 5 - PDF-as-Canvas Architecture

## Agent角色分配

### Manager
- **职责**: 协调所有Agent，分配任务，审查结果，更新项目状态
- **工作重点**: 任务分配、进度跟踪、冲突解决

### Planner
- **职责**: 分析任务，制定详细实施计划，评估技术可行性
- **工作重点**: 技术方案设计、风险评估、最佳实践建议

### Builder
- **职责**: 执行具体的编码和实现任务
- **工作重点**: 功能实现、代码质量、性能优化

### Reviewer
- **职责**: 审查Builder的工作成果，确保代码质量和功能完整性
- **工作重点**: 代码审查、测试验证、质量保证

### Documenter
- **职责**: 维护项目文档，更新状态和日志
- **工作重点**: 文档更新、进度记录、知识管理

## 开发阶段与任务分配

### 阶段1: 修复视觉元素更新问题 (1-2天)

**Planner任务**:
- 分析视觉元素更新问题的根因
- 制定修复方案
- 评估技术可行性

**Builder任务**:
- 修复 `draw_memory_elements` 方法
- 确保内存元素正确渲染在PDF上方
- 测试不同元素大小和位置的渲染效果

**Reviewer任务**:
- 审查修复代码
- 验证视觉元素更新是否正常工作
- 测试不同场景下的渲染效果

**Documenter任务**:
- 更新 DEVELOPMENT_STATUS.md
- 记录修复过程和结果

### 阶段2: 实现元素移动功能 (2-3天)

**Planner任务**:
- 设计元素移动的实现方案
- 评估不同移动策略的优缺点

**Builder任务**:
- 实现元素拖动功能
- 更新内存元素的位置信息
- 确保视觉更新正确显示

**Reviewer任务**:
- 审查移动功能的实现
- 测试元素移动的流畅性和准确性
- 验证边界情况处理

**Documenter任务**:
- 更新 PLAN.md 和 DEVELOPMENT_STATUS.md
- 记录元素移动功能的实现细节

### 阶段3: 实现PDF文本更新 (3-4天)

**Planner任务**:
- 研究PyMuPDF的文本更新功能
- 设计PDF文本更新的实现方案
- 评估性能影响

**Builder任务**:
- 使用PyMuPDF实现PDF文本更新
- 测试不同文本大小和位置的更新效果
- 确保更改在保存时正确持久化

**Reviewer任务**:
- 审查PDF文本更新的实现
- 测试文本更新的准确性和可靠性
- 验证保存功能是否正常工作

**Documenter任务**:
- 更新项目文档
- 记录PDF文本更新的实现细节和测试结果

### 阶段4: 实现LaTeX同步 (3-4天)

**Planner任务**:
- 设计LaTeX同步的架构
- 评估同步策略的可行性
- 制定实现计划

**Builder任务**:
- 实现从内存元素生成LaTeX代码的功能
- 实现LaTeX源文件的更新
- 测试往返编辑（LaTeX → PDF → 编辑 → LaTeX）

**Reviewer任务**:
- 审查LaTeX同步的实现
- 测试同步的准确性和可靠性
- 验证复杂场景下的同步效果

**Documenter任务**:
- 更新项目文档
- 记录LaTeX同步的实现细节和测试结果

### 阶段5: 实现PDF-to-LaTeX重建引擎 (4-5天)

**Planner任务**:
- 设计PDF-to-LaTeX重建引擎的架构
- 评估重建策略的可行性
- 制定实现计划

**Builder任务**:
- 实现从PDF提取元素并重建LaTeX代码的功能
- 测试不同类型元素的重建效果
- 确保重建的LaTeX代码可编译

**Reviewer任务**:
- 审查PDF-to-LaTeX重建引擎的实现
- 测试重建的准确性和可靠性
- 验证复杂文档的重建效果

**Documenter任务**:
- 更新项目文档
- 记录PDF-to-LaTeX重建引擎的实现细节和测试结果

## 协作流程

1. **任务启动**:
   - Manager接收用户任务请求
   - Manager分析任务并分配给Planner
   - Planner制定详细实施计划
   - Manager根据计划分配任务给Builder

2. **开发执行**:
   - Builder执行具体实现任务
   - Builder完成后将结果提交给Reviewer
   - Reviewer审查代码并提供反馈
   - Builder根据反馈进行修改
   - Builder再次提交给Reviewer
   - Reviewer验证通过后通知Manager

3. **文档更新**:
   - Manager通知Documenter更新项目文档
   - Documenter更新相关文档
   - Documenter将更新结果通知Manager

4. **任务完成**:
   - Manager审查最终结果
   - Manager更新项目状态
   - Manager向用户报告任务完成情况

## 通信机制

- **消息格式**: JSON格式，包含id、ts、from、to、type、thread_id、body、meta字段
- **消息类型**: TASK_REQUEST、PLAN_REQUEST、PLAN_RESULT、BUILD_REQUEST、RESULT、REVIEW_REQUEST、REVIEW、DOC_UPDATE_REQUEST、DOC_UPDATED
- **消息流转**: User → Manager → Planner → Manager → Builder → Reviewer → Documenter → Manager

## 质量保证

1. **代码审查**:
   - Reviewer负责审查所有代码
   - 确保代码质量和性能
   - 确保符合项目的设计原则

2. **测试验证**:
   - Builder负责单元测试

   - 确保功能正常工作

3. **文档更新**:
   - Documenter负责及时更新文档
   - 确保文档内容准确反映项目状态

4. **项目管理**:
   - Manager负责进度跟踪和风险管理
   - 确保项目按时完成
   - 处理Agent之间的冲突和问题

## 风险评估

1. **技术风险**:
   - PyMuPDF的文本更新功能可能存在限制
   - LaTeX同步可能面临复杂布局的挑战
   - PDF-to-LaTeX重建可能存在精度问题

2. **时间风险**:
   - 某些功能的实现可能比预期复杂
   - 可能需要更多时间进行测试和调试

3. **质量风险**:
   - 代码质量可能受到时间压力的影响
   - 功能可能存在边界情况处理不当的问题

## 应对策略

1. **技术风险应对**:
   - 提前研究PyMuPDF的功能限制
   - 采用渐进式实现策略，从简单场景开始
   - 优先实现核心功能，复杂功能后续迭代

2. **时间风险应对**:
   - 合理分配任务，避免单一Agent成为瓶颈
   - 建立定期检查点，及时调整计划
   - 预留缓冲时间处理意外情况

3. **质量风险应对**:
   - 严格执行代码审查流程
   - 建立自动化测试框架
   - 注重代码可读性和可维护性
